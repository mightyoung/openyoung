# DI Container 必要性分析

## 当前状态

`src/core/di.py` 实现了一个完整的依赖注入容器：
- Container 类：注册/解析/单例/覆盖
- @inject 装饰器：但目前是 no-op (直接返回原函数)
- 全局容器实例

**使用情况：0 次** - 整个代码库没有任何地方使用它

## 专家视角分析

### 为什么 DI 在某些项目中很重要

1. **解耦** - 模块不直接创建依赖，而是通过注入
2. **测试** - 可以轻松替换为 mock 对象
3. **可替换性** - 同一接口的不同实现可以切换

### 为什么 DI 在这个项目中可能不那么重要

1. **Python 特性** - Python 是动态语言，更倾向于 duck typing
2. **现有模式** - 项目使用其他解耦方式：
   - 全局单例 (`get_xxx()` 函数)
   - 配置对象传递
   - 依赖别名 (`from x import y as z`)
3. **迁移成本** - 现有代码都是直接实例化

## 方案对比

### 方案 A: 启用 DI (成本: 高，收益: 中)

**优点：**
- 更好的可测试性
- 更清晰的依赖关系
- 符合企业级最佳实践

**缺点：**
- 需要大量重构现有代码
- 学习成本
- 可能过度设计

### 方案 B: 保持现状 (成本: 低，收益: 当前)

**优点：**
- 无迁移成本
- 现有模式工作正常
- Pythonic 风格

**缺点：**
- 测试时需要其他 mock 方式
- 依赖关系不够显式

### 方案 C: 仅用于测试 (成本: 低，收益: 中)

**优点：**
- 最小迁移成本
- 特定场景受益
- 不影响现有代码

**缺点：**
- 不够完整
- 可能有不一致感

## 建议

### 结论: 方案 C - 仅用于测试场景

理由：
1. 项目主要使用 Python 风格的解耦（全局函数、配置传递）
2. 大量重构成本高，收益不确定
3. 可以为特定测试场景启用 DI

### 实施建议

如果决定启用，建议：
1. **测试优先** - 只在测试中使用 DI
2. **渐进式** - 新代码使用 DI，旧代码不强制
3. **保持简单** - 不需要复杂的 scope 管理
